# @knowledge-base

OBJETIVO: Crear y mantener una base de conocimiento orientada a resolución que reduzca tickets repetidos y fricción de onboarding.

USAR CUANDO: Hay que escribir, reorganizar o priorizar artículos de help center o documentación de soporte.

NO USAR CUANDO: La respuesta depende de debugging técnico profundo o de información todavía no validada. Primero resolver y luego documentar.

INSTRUCCIONES:

1. Priorizar artículos por volumen de tickets, impacto operativo y fricción de onboarding o activación.
2. Escribir artículos orientados a resolución con problema, pasos, limitaciones, referencias y criterio de escalación.
3. Organizar la base por tareas y lenguaje del usuario final para que el contenido sea encontrable sin conocer la estructura interna del equipo.
4. Cerrar con gobierno editorial: owner, fecha de revisión y disparadores de actualización tras cambios de producto o tickets repetidos.

FORMATO DE SALIDA:

- Problema o pregunta cubierta
- Pasos de resolución
- Limitaciones o cuándo escalar
- Owner y próxima revisión
- Riesgo de obsolescencia o hueco documental

RECURSOS RELACIONADOS:

- Ver `./_playbook.md` para checklist de knowledge base.
- Ver `./_templates.md` para el template de artículo.

ANTI-PATRÓN: Escribir artículos largos sin pasos concretos, organizar la taxonomía para el equipo interno o dejar documentación sin dueño ni revisión tras cambios de producto.
