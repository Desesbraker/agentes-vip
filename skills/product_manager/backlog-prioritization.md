# @backlog-prioritization

OBJETIVO: Ordenar backlog de forma ejecutable usando criterios comparables, visibles y ligados a capacidad real del equipo.

USAR CUANDO: Hay que decidir qué entra ahora, qué queda después y qué no debe romper el foco actual del sprint, release o roadmap.

NO USAR CUANDO: Los items todavía son vagos, no tienen problema definido o dependen de discovery pendiente. Primero se clarifican antes de puntuar.

INSTRUCCIONES:

1. Normalizar cada item con problema, usuario afectado, outcome esperado, esfuerzo estimado y dependencias. Lo ambiguo no se prioriza.
2. Elegir un único framework dominante según el contexto: RICE o ICE para comparables, MoSCoW para releases o handoffs entre equipos.
3. Incluir impacto, urgencia, riesgo, capacidad del equipo y restricciones externas para que la priorización refleje realidad operativa y no solo preferencia estratégica.
4. Cerrar con un orden ejecutable y buckets claros como Now, Next y Later, manteniendo fuera del foco actual todo lo que no está listo.

FORMATO DE SALIDA:

- Framework elegido y criterio
- Items priorizados
- Score o justificación por item
- Buckets Now, Next y Later
- Riesgos, dependencias y bloqueos

RECURSOS RELACIONADOS:

- Ver `./_playbook.md` para checklist de priorización.
- Ver `./_templates.md` para el template de backlog.

ANTI-PATRÓN: Priorizar por jerarquía, por ruido de stakeholders o por promesas vagas sin impacto, esfuerzo y dependencias visibles.
