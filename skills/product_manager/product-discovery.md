# @product-discovery

OBJETIVO: Validar o invalidar hipótesis de producto con evidencia suficiente para decidir si avanzar, iterar, redefinir o matar una iniciativa.

USAR CUANDO: Hace falta entender mejor un problema, un usuario, una oportunidad o una hipótesis antes de escribir un PRD o construir una solución.

NO USAR CUANDO: El problema ya está validado, el scope está acordado y solo falta coordinar implementación. En ese caso usar @product-manager o @backlog-prioritization.

INSTRUCCIONES:

1. Formular una hipótesis explícita de problema, usuario y outcome antes de entrevistar, prototipar o correr experimentos.
2. Elegir el método de validación que mejor responda a la incertidumbre principal: entrevistas, smoke test, prototipo, concierge, fake door o análisis de comportamiento.
3. Sintetizar la evidencia en decisiones: qué se confirma, qué se invalida y qué sigue incierto, distinguiendo señal fuerte de feedback anecdótico.
4. Cerrar con recomendación operativa: avanzar, iterar, matar o redefinir el scope, más los siguientes pasos y la evidencia que aún falta.

FORMATO DE SALIDA:

- Hipótesis evaluada
- Método de validación y muestra
- Evidencia observada
- Decisión recomendada
- Incertidumbres y próximo paso

RECURSOS RELACIONADOS:

- Ver `./_playbook.md` para checklist de discovery.
- Ver `./_templates.md` para el discovery brief.

ANTI-PATRÓN: Hacer discovery para confirmar sesgos propios, mezclar opiniones aisladas con evidencia fuerte o seguir construyendo cuando la hipótesis central ya no se sostiene.
